# 第十三阶段：通用故事孵化器与多作品书库

## 1. 阶段目标

把当前“夜巡人专用故事架构器”改造成适用于任意长篇、短篇的创意入口，并补齐普通用户管理多本小说的完整体验：

```text
打开作品库
→ 新建长篇或短篇
→ 与创意 Agent 多轮讨论
→ 固化 StoryBrief
→ 生成并审查 Canon 提案
→ 生成分层规划与下一批章节节拍
→ 锁定后进入试写
```

“夜巡人”继续保留，但只作为可选示例模板，不再作为所有新作品的默认业务规则。

## 2. 当前真实状态

已经具备：

- 作品目录库与多项目切换。
- 每部作品独立目录、`story.db`、Canon、规划、正文、审计和备份。
- 新建正式项目从第 0 章开始，业务数据不依赖浏览器 `localStorage`。
- Canon 草稿、分析、提案、锁定与变更申请基础。
- 长篇章节生产、质量门、自动托管、导出和耐久测试基础。
- 短篇策略与独立短篇项目生产后端基础。

尚未具备：

- 面向任意题材的多轮创意访谈与会话持久化。
- 从对话中形成可版本化、可确认的通用 `StoryBrief`。
- 根据长篇/短篇、目标字数和题材动态生成 Canon 与规划。
- 当前 Phase 8 中仍存在夜巡人、1000 章、七卷、沈砚、雾城等固定决策。
- 普通用户可操作的短篇策略与物化向导 UI。
- 完整的作品归档、复制、恢复入口和历史筛选体验。

## 3. 多作品书库

完善 `/overview` 为正式作品库：

- 卡片展示作品类型、题材、当前章节、Canon 状态、最近打开时间、自动托管状态和本地目录。
- 支持搜索、按长篇/短篇筛选、最近使用排序。
- 支持新建、打开、重命名、复制、归档和从备份恢复；删除必须二次确认并默认只做归档。
- 新建时选择：空白长篇、空白短篇、从模板开始、从现有作品复制设定。
- 项目间不得共享 SQLite 主键、正式正文、Canon revision、模型会话或费用记录。
- 有正式项目时不默认打开 demo；demo 明确标记且不能直接运行付费自动托管。

## 4. 创意工作室

新增正式路由 `/ideation`，保持“墨曜指挥舱”视觉体系和固定右侧 AI 控制台。

### 4.1 多轮访谈

用户可以只输入一句模糊想法，例如“想写现代悬疑，篇幅不要太长”。创意 Agent 应根据缺口逐轮追问，而不是一次性替用户拍板。

访谈至少覆盖：

- 作品形态：长篇或短篇、目标章节/总字数、单章字数。
- 类型、受众、平台倾向、叙事视角、基调和节奏。
- 核心创意、主题、主角欲望、核心冲突和结局方向。
- 世界边界、时代地域、能力/职业/资源体系。
- 感情线强度、禁写内容和内容安全边界。
- 参考作品只抽取抽象特征，不复制人物、情节、专有名词或原文文风。

AI 每轮展示“已确认、待确认、AI 建议、相互冲突”四类信息。用户可以直接编辑，也可以要求 AI 给出 2—3 个备选方向。

### 4.2 数据权威

新增持久化对象：

- `IdeationSession`：所属作品、状态、revision、创建/更新时间。
- `IdeationMessage`：角色、内容、模型运行、作用域和时间。
- `StoryBriefVersion`：完整结构化 Brief、来源会话、版本、checksum 和状态。
- `StoryBriefProposal`：修改前后差异、理由、影响和确认状态。

对话不是正式设定。只有用户点击“固化为 StoryBrief”并确认差异后，新的 Brief 版本才成为 Canon 生成输入。任何模型回复不得直接修改正式 Canon 或规划。

## 5. 通用 Canon 架构器

移除生产路径中的夜巡人固定常量，改为由已确认 `StoryBriefVersion` 驱动：

- 故事内核、世界边界、时代与地域。
- 人物、人物知识边界、关系和组织。
- 物品、资源、技能/能力/职业体系、升级条件、限制与代价。
- 世界硬规则、冲突规则和异常处理方式。
- 主线、支线、伏笔、揭示窗口和禁止提前完成事项。
- 文风、叙事视角、节奏、单章结构和禁写约束。

确定性检查器只检查通用结构与交叉一致性，不假设作品必须有等级、法宝或怪异规则；不存在的体系标记为“不适用”，存在时才强制检查完整性。

Canon 仍采用“生成提案 → 差异审查 → 接受/拒绝 → 锁定”流程。锁定后只能走变更申请，且 revision 冲突必须返回 409。

## 6. 动态长篇与短篇规划

- 长篇根据目标章节动态计算卷、故事弧和里程碑，不固定 1000 章或七卷。
- 短篇根据目标总字数和章节数生成紧凑节拍，不强制存在卷结构。
- 全局只生成分层骨架；精确章节节拍维持“下一批 5 章”滚动细化。
- 继续使用最早/目标/最晚章节、前置条件、完成条件、人物最早出场、能力最早获得、真相揭示窗口等预算台账。
- 每章契约必须绑定当前 `ChapterBeat`；未来节拍继续进入禁止提前消费边界。
- “夜巡人”模板可预填现有六阶能力、四级法器和七卷基线，但用户必须主动选择。

## 7. 短篇生产入口

在作品库和创意工作室提供两种路径：

1. 原生短篇：从空白 Brief 直接生成独立 `short-form` 项目。
2. 长篇改写短篇：选择来源长篇、冻结来源 manifest、生成压缩策略，再物化为独立短篇项目。

补齐 Phase 11/12 后端对应 UI：来源范围、目标篇幅、核心钩子、保留/删除内容、策略 findings、提案差异、物化状态和失败恢复。短篇仍复用现有章节生产、质量门、自动托管与导出链路。

## 8. API 预设计

- `POST /api/v1/projects/{project_id}/ideation/sessions`
- `GET /api/v1/projects/{project_id}/ideation/sessions`
- `POST /api/v1/ideation/sessions/{session_id}/messages`
- `POST /api/v1/ideation/sessions/{session_id}/story-brief-proposals`
- `GET /api/v1/projects/{project_id}/story-brief/versions`
- `POST /api/v1/story-brief-proposals/{proposal_id}/apply`
- `POST /api/v1/story-brief-proposals/{proposal_id}/reject`
- `POST /api/v1/projects/{project_id}/archive`
- `POST /api/v1/projects/{project_id}/duplicate`

所有写操作携带 expected revision；模型调用在事务外执行；保存结果前重新检查项目和会话 revision。

## 9. 验收标准

- 同时创建至少 2 本长篇和 2 本短篇，重启服务后均可独立打开，数据不串联。
- 从一句模糊想法完成至少 3 轮对话，形成 StoryBrief 提案；拒绝不改变权威状态，接受后可恢复版本历史。
- 分别生成非怪谈题材长篇、无等级体系短篇，结果中不得出现夜巡人、沈砚、雾城、六阶或七卷残留。
- StoryBrief、Canon、Plan 三层 revision 漂移均能阻止旧模型结果落盘。
- 长篇和短篇规划范围、总字数与章节预算一致。
- 夜巡人模板仍能生成现有基线，但只能由用户主动选择。
- 短篇策略/物化 UI 可完成提案、确认、独立项目创建和失败恢复。
- API、Web、Build、Playwright 两档分辨率全部通过。
- 测试使用确定性 Provider；真实 Provider 只做一次人工冒烟，不提交密钥、数据库或生成正文。

## 10. 实施顺序与协作边界

1. 后端先实现会话、StoryBrief 版本和通用生成契约。
2. 当前电脑实现作品库与创意工作室 UI，保持既有视觉令牌。
3. 移除 Phase 8 生产路径硬编码并迁移夜巡人为模板。
4. 接入 Canon/Plan 提案闭环。
5. 补齐短篇策略与物化 UI。
6. 全量测试后再开放任意新书真实付费试写。

另一台电脑可以实现第 1、3 项后端，但不得修改 `apps/web/**`、CSS、设计令牌或视觉快照；UI 始终由当前电脑完成和验收。
